본 문서는 VPC 구축 후 복습 용도로 학습한 내용을 정리, 리뷰를 담은 문서입니다.

1. VPC를 구축해야 하는 이유

처음 AWS 를 이용해 WEB/WAS 서버, DB 서버 구축시 AWS 계정당 기본적으로 있는 Default VPC 에 서버를 구축했었다.
WEB/WAS 는 하나의 EC2 에서 관리하고, DB 는 Default VPC 의 RDS 로 관리하였다.
당시 CI/CD 파이프라인은 Github Actions 를 이용하여 직접 Default VPC 의 EC2에 접근하여 배포하는 방식을 사용했다. 이것은 퍼블릭, 즉 인터넷이 되는 곳이라면 어디서든 내 기본 VPC 에 접속정보만 탈취하면 접근 가능하다는것을 의미했고, 이는 보안에 매우 취약하다는것을 의미한다.

그럼 Default VPC는 왜 보안에 취약한가?
Default VPC 는 기본적으로 인터넷 접근이 가능한 퍼블릭 구조로 되어있다. 즉 기본 VPC에 존재하는 서버는 퍼블릭IP 를 쉽게 가지게 되고 외부 접근 가능성이 높다.
이는 기존 CI/CD 파이프라인에서 Gtihub Actions 에서 명령어만으로 EC2 에 접근하여 배포할 수 있다는것으로 보안 취약점이 증명되었다.

또한 웹/앱서버를 통합하여 운영하여 역할의 단일화가 구성되어있지 않아 서버에 오류가 발생하면 웹서버의 오류인지, 앱서버의 오류인지 파악하는데 시간이 걸렸다.
그럼 3-Tier 아키텍처를 구현하면 되지 않느냐? 라고 질문할수도 있지만, 기본 VPC의 경우 모든 서브넷이 외부에서 접근가능한 인터넷 게이트웨이와 연결되어있고, 인스턴스 생성시 자동 퍼블릭 IP를 할당하기 때문에 프라이빗 서브넷을 요구하는 3-Tier 아키텍처 구현이 기본 VPC 에는 불가능하다.

이러한 이유 때문에 기본적으로 VPC 를 구축하고 3-Tier 아키텍처로 구축하는게 AWS 아키텍처의 표본이자 제일 많이 사용하는 구조라는것을 알게 되었고 VPC를 구축하여 기존 웹, 앱서버가 통합되어 있는 EC2 를 웹서버 용, 앱서버 용 EC2로 각각 분리하여 VPC 내 각 프라이빗 서브넷에 구현해두었고, DB 는 DB 프라이빗 서브넷에 구축하였다.

따라서 내가 구축한 AWS VPC 환경은 다음과 같다.

VPC
├ Public Subnet
│ └ ALB
│
├ Private Subnet
│ ├ Web Server
│ └ App Server
│
└ DB Subnet
└ RDS

2. 3-Tier-아키텍로 구축해야 하는 이유는?

일전에 기본 VPC 에서 EC2 하나테 웹/앱서버를 같이 구축하였다. 웹서버, 앱서버를 통합하여 관리할 경우 필자가 겪은 문제점은 다음과 같다.

1. EC2 가 죽으면 웹,앱서버가 모두 불능상태가 된다 => 이는 고가용성은 불가하고 내결함성도 만족하지 못하는 네트워크이다.
2. 웹 or 앱서버에 문제 발생시 어디서 문제가 발생했는지 원인파악시 많은 시간 소모
3. 필요 기능 or 타 EC2로 확장에 불리함 => 확장성 저하
4. 웹, 앱서버의 통합관리로 인해 시스템 변경시 타 프로그램에 영향을 끼칠 가능성이 있음

따라서 이러한 단점을 보완한 3-Tier 아키텍처를 구현하기로 했다.
아키텍처 구조는 다음과 같다.

Public Subnet
└ ALB

Private Subnet
├ Web Server
└ App Server

DB Subnet
└ RDS
이렇게 각 역할에 따라 분리하여 보안, 확장성, 안정성을 동시에 확보할 수 있다.

3. 고가용성, 내결함성이 뭔데?
   고가용성 : 고가용성은 시스템 내부에 장애가 발생해도 성능이 떨어지면 안된다. 시스템상 장애가 발생하더라고 서비스 중단 시간을 최소화하여 지속적으로 서비스를 제공하는 능력을 의미함
   내결함성 : 시스템 일부가 장애가 발생하더라도 서비스가 중단되지 않고 계속 정상적으로 동작하도록 설계하는 능력을 의미함

어떻게 보면 고가용성이 더 상위 개념으로 볼 수 있다.

내가 이것을 보고 느낀점은 이것이었다.

고가용성, 내결함성은 무슨의미인지 알겠어, 근데 어떻게 만들어야하는데?

고가용성을 만족하는 방법
고가용성을 만족하기 위해선 다음과 같은 특징들이 네트워크로 구현되어있어야 한다.
이중화 : 여러 서버를 준비하여 하나가 장애 나도 다른 서버가 서비스 제공
자동복구 : 장애 발생 시 자동으로 새로운 인스턴스 생성
로드밸런싱 : 트래픽을 여러 서버에 분산

내결함성을 만족하는 방법
완전한 서버 이중화, 다중화
실시간 동기화
장애 발생시 즉시 대체
등이 있다.
